## 2. 変更前提ルール

- 未指示の変更は行わない。必要な追加変更は提案し、合意後に実施する。
- UI/UX変更（レイアウト・色・フォント・間隔・操作導線）は、事前合意を得てから実施する。
- 技術スタックのバージョンは勝手に変更しない。必要時は理由を示して合意を得る。

## 3. タスク完了時の必須更新

UI/UX・機能・運用に関わる変更を行ったら、同ターンで必ずドキュメントも更新する。

- `.cursor/docs/uiux-improvement-log.md`
  - チェックリストを更新（完了は `[x]`）
  - ブレスト中で確定した項目はチェックリストへ移動
  - 実装済みが「未確定/ブレスト中」に残っていたら削除
  - 実装ログは必ず「新しいものを上、古いものを下」
- リファクタ関連の変更時は `.cursor/docs/refactoring-checklist.md` も更新
- E2E/テスト基盤変更時は `.cursor/docs/e2e-test-integration-checklist.md` も更新
- コード変更後は `.cursor/docs/project-overview.yaml` の更新要否を確認し、必要なら反映する

## 4. 実装・進行ルール

- ユーザーの直近指示を最優先する。未依頼の別タスクに逸れない。
- 既存UIとの統一感を維持する（サイズ、余白、文言トーン、操作導線）。
- 高リスク変更（認証、Firebaseルール、データ整合性）は根拠を確認してから実施する。
- 設定系は `src/lib/firebaseClient.ts` 一元管理方針を守る。
- データ構造/権限モデル変更時は、`firestore.rules` とアプリ実装の整合を必ず確認する。
- Firebaseの権限はサーバールールに加えて、クライアント側でも対象データ所有チェックを行う。

## 5. 実装後レビュー参照

- 実装後セルフレビューの詳細項目は `.cursor/docs/uiux-improvement-log.md` の「反映メモ（運用）」を正本として参照する。

## 6. コーディング要点

- ディレクトリ責務を守る（`src/pages`, `src/components`, `src/hooks`, `src/lib`, `src/schemas` など）。
- バリデーションスキーマは `src/schemas` に集約し、型はスキーマ由来を優先する。
- 外部データ取り込み時は `src/lib/type-guards.ts` などで型安全な変換を通す。
- スタイル実装は Tailwind CSS を原則とし、インライン `style` は Tailwind で表現不能な場合のみ使用する。
- インライン `style` を使った場合は、理由を明示し、Tailwind へ戻せるなら残タスク化する。
- CSSでベンダープレフィックスを使う場合は、対応する標準プロパティを必ず併記する。

## 7. コミット運用

- コミットは機能単位で分割する。
- 1コミット1目的を原則とし、UI変更・データ仕様変更・ドキュメント更新を必要に応じて分離する。
- 未コミット差分が混在している場合は、差分を整理してから提案・実施する。
- `gh pr create` / `gh pr edit` の本文指定では、シェル展開事故を防ぐため `--body-file` を優先し、`--body` を使う場合はシングルクォートで囲んでバッククォートを使わない。
- コミット前に最低限 `npm run type-check` を実行する。

## 8. 重複ドキュメント方針

- 運用記録の正本は `.cursor/docs/*` に統合する。
- 役割が重複するファイルは片方を案内用にし、正本へのリンクを明記する。

## 9. Geminiレビュー運用

- コード変更がある作業では、実装前に「この変更でGemini差分レビューを行うか」をユーザーへ短く確認する。
- Gemini CLI の差分レビューは、`ユーザーが明示指示した時` または `コミット前/PR前` に実行する。
- 通常の実装ターンでは毎回実行しない。中間状態での反復レビューは原則行わない。
- レビュー実施時の手順は `gemini-pr-review` Skill を使用する。
- 最低限 `git diff --no-color` を Gemini に渡し、High/Medium/Low の指摘を確認する。
- High 指摘がある場合は、修正してからコミット/最終報告するか、未対応理由を明記して報告する。

## 10. UI変更時の表示QA運用（必須）

- UI変更時の表示確認は `Chrome MCP` を第一候補とし、対象画面へ自動遷移して `iPhone系` と `Desktop` の2種類でスクリーンショットを取得する。
- `Chrome MCP` が失敗した場合は、同一ターン内で `Playwright` に即フォールバックして同じ2viewportの取得を試みる。
- スクリーンショットは、対象画面の読み込み完了を確認してから取得する。最低限 `URL一致` に加えて `主要UI要素（例: 見出し/フォーム/カード）` の表示確認を満たすまで待機する。
- 開発環境の起動が遅い場合は、待機時間を延長して再確認する。読み込み未完了状態のスクリーンショットは成果物として扱わない。
- 変更前後の画像パスと、差分確認結果（何が変わったか / 期待どおりか）を必ず報告する。
- 上記 2viewport の取得と報告が揃うまで、作業を「完了」としない。
- `webkit` が環境依存で起動失敗する場合は、`chromium` のモバイルエミュレーション（iPhone系）+ `chromium` Desktop を代替とし、失敗ログ要点を添えて報告する。
- `Chrome MCP` と `Playwright` の両方で取得に失敗した場合は、失敗ログ要点・再現条件・次の打ち手を示し、ユーザー合意なしに完了報告しない。
- プロセス停止は「このターンで自分が起動した対象プロセス」に限定する。既存稼働中の開発サーバーや他タスクのプロセスは停止しない。

## 11. 事前調査・失敗時運用（全タスク共通）

- 何かを実行する前に、`Gemini CLI` で「今の環境条件（OS/実行方式/tmux/制約）」を含めて実行方法を先に調査し、その結果に沿って実行する。
- 実行に失敗した場合は、失敗ログを保持し、同じ条件での再試行前に Gemini へ追加調査を行ってから次の手順を選ぶ。
- 複数回（2回以上）失敗した作業は、以後の同種タスクでもこの `Gemini事前調査 -> 実行 -> ログ確認` を標準運用として適用する。
- 上記運用を行ったことは、最終報告で「調査に使ったGeminiプロンプト要点」と「採用した実行手順」を簡潔に明記する。

## 12. 自律的な機能拡充運用（全タスク共通）

- ユーザーの個別指示がなくても、着手前に一次情報（公式ドキュメント）と実践記事を調査し、現在の環境で再現可能な運用を選定する。
- 「記事や公式ができると述べる内容」と「この環境で実際に実行できる内容」を毎回比較し、差分（未達・不安定・要追加設定）を明示する。
- 差分がある場合は、設定改善・検証追加・運用変更を自律的に提案し、承認後に実施する。
- 実施後は再現可能性を優先し、実行コマンド、保存物パス、成功/失敗ログ要点を最終報告に残す。
